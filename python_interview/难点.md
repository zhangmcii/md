总结：
这个项目整体实现不复杂，但有几个比较实际的难点。
一个是数据结构的问题，原始数据是嵌套的 JSON，但页面编辑和导出结构不一致，比如导出是按分镜组拆分的，所以需要做一层数据重组，这里处理起来比较容易出错。
第二个是编辑和状态的联动，比如数据标记为“已确认”后，如果用户再修改，需要自动回退状态，这里需要统一处理数据变更，而不是在每个字段里写逻辑。
另外一个是防止数据丢失，因为标注过程比较长，我加了本地缓存，避免刷新后数据丢失。
还有一个考虑是没有把系统做复杂，比如没有引入数据库，而是保持后端轻量，这其实也是一个取舍。

1. 实时通信与状态一致性 (Socket.IO + Vuex/Pinia)
潜在难点： 即使你觉得 Socket.IO 用起来很简单，但如何在高频社交场景下保证前端状态与后端数据的一致性是有深度的。

你可以这样说： “项目集成了实时聊天功能。难点在于如何处理网络波动导致的 Socket 断连与重连。如果直接重连，可能会丢失断线期间的消息。我研究并实现了心跳机制（Heartbeat）和消息确认（ACK）机制，确保消息不丢失。同时，在前端我需要将 Socket 推送的数据实时合并到本地的 Pinia 状态中，并处理好消息去重，防止因为网络抖动导致同一条消息显示两次。”

2. 鉴权体系的安全性与统一 (OAuth + JWT + Socket)
潜在难点： 你的项目有 OAuth 登录。如何让 HTTP 请求和 WebSocket 连接共享一套鉴权逻辑？

你可以这样说： “项目中使用了 OAuth 第三方登录和 JWT。难点在于长连接（Socket.IO）的权限校验。普通的 API 请求可以在 Header 带 Token，但 WebSocket 的握手阶段比较特殊。我需要实现在 Socket 建立连接时进行中间件校验，并在 Token 过期后，如何平滑地让长连接也完成无感刷新，而不是直接把用户踢下线。”

3. 移动端优先的体验与“优雅降级” (Graceful Degradation)
潜在难点： 你的 README 提到了“优雅降级”，这本身就是一个非常高级的面试话题。

你可以这样说： “作为一个移动端优先的项目，我重点处理了复杂交互在低端机型或弱网环境下的表现（优雅降级）。例如，在网络极差导致 Socket 无法连接时，我设计了一套降级方案，让实时聊天功能回退到短轮询（Short Polling）或者至少保证基础的博客浏览不受影响。此外，针对移动端的视口抖动、键盘遮挡输入框等细节，我也做了深度适配，这比单纯实现功能要花更多精力去调试。”

4. 数据模型与性能考量 (Flask + SQLAlchemy)
潜在难点： 社交项目涉及“用户-文章-评论-点赞”这种多对多的复杂关系。

你可以这样说： “虽然功能写完了，但在设计数据库模型时，我考虑到了社交关系的查询效率。比如‘点赞’和‘关注’，如果直接 Join 多个大表，性能会随着数据量增加而下降。我通过优化索引，以及在某些地方采用‘空间换时间’的思路（冗余字段）来减少复杂查询。此外，针对 Flask 这种同步框架处理异步 Socket 的并发压力，我通过引入 Eventlet 协程库进行了优化，保证了服务器在高并发下的稳定性。”

## 技术深挖（并发与实时性）
WebSocket 扩展性： 
Flask 本身是 WSGI 协议，它是同步的。你在实现 Socket.IO 时，底层使用的是线程模式还是协程（如 Gevent/Eventlet）？如果现在有 5000 个用户同时在线保持长连接，你的服务器内存和 CPU 占用大概在什么水平？你如何监控并定位其中的死锁或资源泄露问题？

状态同步一致性： 
在社交场景下，如果 A 用户点赞了 B 用户的文章，此时 B 正在通过 WebSocket 聊天。你是如何保证点赞动态能即时、准确地推送到 B 的界面，同时保证数据库里的计数和前端显示的数字在网络波动时依然保持最终一致性？

身份校验的成本： 
你的项目使用了 JWT 和 OAuth。对于 WebSocket 连接，你是在握手阶段校验一次权限，还是在每次消息传输时都进行校验？如果是前者，如何处理 Token 在连接期间过期的情况？如果是后者，如何在高频通信中降低鉴权带来的性能损耗？

## 数据建模与存储（深度与优化）
关系模型性能： 
社交系统最核心的是“关系”（如：粉丝、关注、好友）。在你的数据库设计中，如果一个“大 V”拥有 100 万粉丝，当你查询他的粉丝列表或者判断某个用户是否是他的粉丝时，你的 SQL 语句是如何优化的？有没有考虑到深度分页带来的性能坍塌？

复杂嵌套处理： 
博客系统必然涉及“多级评论”或“嵌套回复”。你是采用递归查询（Recursive Query）、路径枚举（Path Enumeration）还是闭包表（Closure Table）来存储这种树形结构？请从查询效率和写入复杂度的角度对比说明你的选择理由。
