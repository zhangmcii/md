Flask里有两个核心表：
```
# URL规则
url_map: Rule -> endpoint

# endpoint映射
view_functions: endpoint -> view_func
```

URL可以变化，但逻辑不变

url_for 的关键问题, url_for 是 endpoint → URL.如果你只有：URL -> view_func。
他会
```
遍历所有URL
找到view_func匹配的
再生成URL
```

允许一个函数绑定多个URL



解耦。

URL是路由层
endpoint是逻辑标识
view_func是实现

2."user_detail" Rule2 -> "user_detail" Rule3 -> "user_detail" endpoint -> user_detail 可endpoint是唯一的，这样写不冲突吗?

endpoint 确实必须唯一。但唯一的是 endpoint → view_func 这层，不是 Rule → endpoint。
总结一下这个机制:
| 层         | 是否唯一           |
| --------- | -------------- |
| Rule      | 可以多个           |
| endpoint  | 必须唯一           |
| view_func | 一个endpoint对应一个 |



3.大型系统里常见到手动指定endpoint，什么情况下需要这样做？

| 场景            | 原因       |
| ------------- | -------- |
| API版本         | 需要多个稳定名字 |
| 多URL同函数       | 需要区分反向路由 |
| Blueprint复杂项目 | 避免命名冲突   |
| 类视图           | 没有函数名    |
| 插件开发          | 保证全局唯一   |
| 大型重构          | 函数名可以随便改 |


## 代理和LocalProxy
为什么Flask需要使用LocalProxy代理？
总体来说，这是因为代理对象可以在线程间共享，使我们能够以动态的方式获取被代理的实际对象。

具体来说,当上下文未被激活时，响应的全局代理对象处于未绑定状态。如果不使用代理，在导入这些全局对象时会尝试获取上下文，此时获取到的全局对象只能是None。当请求进入并调用视图函数时，尽管上下文变量已经被设置，但此前导入的全局对象仍然是None。如果使用代理，则可以动态获取上下文对象。
