## 请求的进入
Flask中的WSGI程序为什么放在单独的方法中实现，而不是直接在__call__()方法中实现？
主要是为了方便在附加中间件的同时保留对程序实例的引用。
中间件会包住 WSGI app，如果逻辑写死在 __call__，原始 Flask 实例就容易“丢失”。所以让中间件只包 WSGI 层，而不替换 Flask 实例本身。

## 代理和LocalProxy
为什么Flask需要使用LocalProxy代理？
总体来说，这是因为代理对象可以在线程间共享，使我们能够以动态的方式获取被代理的实际对象。

具体来说,当上下文未被激活时，响应的全局代理对象处于未绑定状态。如果不使用代理，在导入这些全局对象时会尝试获取上下文，此时获取到的全局对象只能是None。当请求进入并调用视图函数时，尽管上下文变量已经被设置，但此前导入的全局对象仍然是None。如果使用代理，则可以动态获取上下文对象。

## 上下文
Flask中为什么要引入程序上下文，只保留一个请求上下文不就行了吗？
理论上可以。现实里会变成一堆不必要的限制和混乱。

关系是：
```
RequestContext
    └── AppContext
```
请求上下文依赖程序上下文，但程序上下文可以单独存在。

1.有些操作根本没有 HTTP 请求
Flask 自带 CLI命令，比如flask shell, flask db migrate。在这些命令对应的逻辑中，你可能这么写：
```
from flask import current_app

print(current_app.config["DB_URI"])
```
那么问题来了，这里没有HTTP请求，但是你仍然需要，current_app, g, config。如果Flask只有请求上下文，那 CLI 就没法用这些对象。


2.数据库连接生命周期
flask常见的写法：
```
from flask import g

def get_db():
    if "db" not in g:
        g.db = connect_db()
    return g.db

@app.teardown_appcontext
def close_db(error=None):
    db = g.pop("db", None)
    if db:
        db.close()
```
原因很简单：
数据库连接属于 应用资源，不是 HTTP 请求资源


如果放在 RequestContext：CLI, 后台任务, 脚本  这些地方都无法复用。

3.请求对象和应用对象是两种完全不同的东西
请求上下文包含：
```
request
session
url_adapter
environ
```
这些 只在 HTTP 请求存在时才有意义。


程序上下文包含：
```
current_app
g
app.config
extensions
```
这些是 应用级别资源。

如果只用 RequestContext，会出现:为了访问 current_app, 必须伪造 HTTP 请求

4.蓝图和扩展需要应用上下文
很多的扩展会这么写：
```
from flask import current_app

def get_cache():
    return current_app.extensions["cache"]
```
这里依赖：current_app, 但是不依赖request、session， 
如果没有 应用上下文， 这些代码只能在 HTTP 请求中运行， 后台任务、CLI 全废。


如果 Flask 只有 RequestContext，会出现三个问题：
| 问题                 | 结果     |
| ------------------ | ------ |
| CLI无法使用current_app | 很多命令会坏 |
| 后台任务无法使用g          | 资源管理混乱 |
| 应用资源和请求资源混在一起      | 生命周期错误 |
